# ION Semantic Model — v0.5 → v0.6 → v1.0 Transition Plan

How ION's vocabulary-publishing story evolves across the next two major versions. Referenced from `schema/core/v2/api/v2.0.0/ion.yaml → x-ion-semantic-model.transitionPlan`.

## v0.5.2-draft — **current state**

**What's true today.**
- OpenAPI schemas (`attributes.yaml`) are the authoritative source of truth for wire validation.
- JSON-LD contexts (`context.jsonld`) are load-bearing as published vocabulary reference, auto-regenerated from OpenAPI via `tools/regenerate_contexts.py`.
- Drift between the two is a build-time warning (`DRIFT_*` in `tools/validate.py`).
- Every pack has a canonical URL at `https://schema.ion.id/{layer}/{pack}/{version}/` — generated by `tools/build_schema_site.py`.
- Allowed vocabulary namespaces are whitelisted: `ion:` and `beckn:` core only.
- Cross-network mappings to `ondc:`, `schema.org:`, or Beckn domain schemas are NOT published.

**What NPs do.**
- BPPs and BAPs validate messages using `attributes.yaml` (OpenAPI) — same as before.
- Every `*Attributes` payload carries `@context` + `@type` as ION-mandatory — unchanged from v0.5.
- When building internal tooling that needs to reason about ION data as a graph, NPs can resolve `https://schema.ion.id/{path}` for canonical vocabulary definitions.

**What's *new* in v0.5.2.**
- schema.ion.id is published (treat as production infrastructure).
- `x-ion-semantic-model` block in `ion.yaml` declares URL patterns, content negotiation, and allowed contexts.
- `DRIFT_*` warnings in the validator catch any future divergence between OpenAPI and JSON-LD.

## v0.6 — **targeted changes**

**What flips.**
- `DRIFT_*` warnings escalate to errors. Any PR that introduces schema↔context drift fails CI.
- Full source-of-truth inversion considered (not yet decided — see trigger below). If committed, `attributes.yaml` becomes a generated artefact; the vocabulary file (Turtle or JSON-LD `@graph`) becomes the authoritative source.
- ION Central's schema gate starts enforcing the `allowedContexts` whitelist. Messages carrying unknown context prefixes are nacked.
- First cross-network mapping published — candidate: ION ↔ ONDC logistics service levels. Published at `https://schema.ion.id/mappings/ondc-logistics/v1/`.

**Trigger for inversion decision.**
The source-of-truth inversion is only taken up if **all three** are true:
1. Upstream Beckn v2.0.0 final has shipped and stabilized.
2. ION has 3+ months of production cross-network traffic data.
3. There is concrete demand from at least one external consumer (AI agent, graph-DB, multi-network aggregator) for a dereferenceable vocabulary beyond what current JSON-LD artefacts provide.

If any of these is false, inversion stays deferred and v0.6 focuses on operational hardening of the existing dual-source setup.

## v1.0 — **target state**

**What's true at target.**
- ION is a first-class JSON-LD-native network. schema.ion.id is machine-discoverable, resolvable, versioned, and mapped to at least one peer network (ONDC).
- AI agents and analytics platforms consume ION data via linked-data tooling without bespoke integration.
- Cross-network reasoning is possible at message time when both networks publish compatible contexts.
- ION participates in cross-country Beckn-family governance discussions as a vocabulary publisher, not a vocabulary consumer.

## What's deliberately NOT promised

- ION does NOT promise schema.beckn.io domain-schema imports. ION's sector taxonomy (13 trade spines, 6 logistics spines) differs deliberately from Beckn's module classification. See Q5.
- ION does NOT promise full SPARQL endpoint or RDF triple store at schema.ion.id. Static JSON-LD + OpenAPI + HTML is the published surface. Reasoning tools run client-side.
- ION does NOT promise automated vocabulary alignment with every other network. Each mapping is a Council-reviewed, explicitly-published artefact.

## Cost & operational commitments

**v0.5.2 operational cost:**
- Static hosting for schema.ion.id — a few dollars a month.
- TLS certificate auto-renewal.
- CI job to rebuild the site on every `main` merge.

**v0.6 operational cost increment:**
- CDN with proper `Accept`-header content negotiation.
- Uptime monitoring (URL stability is a Council-ratified commitment).
- First mapping authoring, review, and publication workflow.

**v1.0 operational cost increment:**
- SLA on schema.ion.id uptime (recommend 99.9% as it becomes network-critical).
- Formal deprecation policy with minimum 12-month overlap between versions.
- Mapping working group + ratification cadence.

## What a third party sees

By dereferencing `https://schema.ion.id/logistics/offer/v1/`:

| Accept header | Response |
|---|---|
| `text/html` | Human-readable reference page with schema tables, README, IRIs |
| `application/ld+json` | `context.jsonld` — term IRI map |
| `application/yaml` | `attributes.yaml` — OpenAPI schemas |
| `application/json` | `context.jsonld` (same as JSON-LD) |

In v0.6+ add:

| Accept header | Response |
|---|---|
| `text/turtle` | `vocab.ttl` — RDF vocabulary definitions |
| `application/rdf+xml` | RDF/XML rendering of same |

## Review milestones

- **v0.5.2 GA** (when Beckn v2.0.0 final ships): re-validate the spec against final Beckn and publish schema.ion.id with production URLs.
- **6 months post-v0.5.2 GA**: first operational review. Is schema.ion.id being used? By whom? Any URL stability issues?
- **12 months post-v0.5.2 GA**: Council review of inversion triggers (the three conditions listed above). Decision on v0.6 scope.
- **24 months post-v0.5.2 GA**: v1.0 scope proposal. Expected to include first ratified cross-network mappings.
